# LLM 代码审计系统 — 使用说明

## 系统概述

本系统是一个可配置的多阶段代码漏洞检测管道，支持 C/C++ 源码的自动审计。核心流程：

```
源代码 → CodeSlicer静态切片 → 静态决策层 → [代码窗口] → LLM分析 → 后处理 → 判定结果
```

所有管道参数均可在 Streamlit 面板中独立调节，无需修改代码或配置文件。

## 二、样本选择

| 参数 | 说明 |
|------|------|
| 数据集 | 5 个测试集可选，详见下表 |
| 样本序号 | 当前数据集中的第 N 个样本 |
| 真实标签 | 数据集中标注的真实结果（有漏洞/安全），用于对比验证 |

### 数据集特征

| 数据集 | 漏洞类型 | 典型规模 | 特点 |
|--------|---------|---------|------|
| BigVul | Buffer overflow, CWE-119/120 | ~1900 | 多为 sink 型漏洞（strcpy/memcpy 等） |
| Juliet | 合成测试用例 | ~2000 | 有精确的 good/bad 对照，sink 明确 |
| D2A | UAF, double-free, 竞态 | ~1500 | **逻辑错误为主，经常无 sink 函数** |
| PrimeVul | 逻辑错误 + 数值漏洞 | ~200 | **off-by-one、整数溢出，常无 sink** |
| 自建集 | 自定义 | 视构造 | 测试时灵活使用 |

## 三、管道配置参数详解

### 3.1 预设方案

快速切换三套预定义配置，也可选「自定义」手动调节每个参数。

| 预设 | LLM 模式 | 调用次数 | Token 消耗 | 适用场景 |
|------|---------|---------|-----------|---------|
| v1 (IRIS) | Agent链 | 2-3 | 中 | 保守策略，宁可误报不遗漏 |
| v2 (多温度投票) | 投票 | 4-5 | 高 | 追求准确性，用多视角降低单次偏差 |
| v3 (单次调用) | 单次 | 1 | **最低** | 快速测试，省 token 省时间 |

---

### 3.2 静态决策层

**作用：** 在调用 LLM 之前，用纯规则判断代码是否明显安全或明显有漏洞。命中则直接返回结果，**零 LLM 调用**。

| 参数 | 可选值 | 默认值(v3) | 说明 |
|------|--------|-----------|------|
| 无sink函数 → | 判安全 / 送LLM | 判安全 | 代码中没有如 strcpy/system 等危险函数时的行为。**判安全**省 token，但 D2A/PrimeVul 的逻辑错误会被漏掉 |
| 低风险+sink → | 判安全 / 送LLM | 判安全 | sink 风险等级为 low 时的行为。**判安全**省 token |
| 消毒器阈值 | 0 ~ 5 | 2 | 检测到 ≥N 个消毒器（如 sizeof 检查、长度校验）时直接判安全。0 = 不启用 |
| 数值漏洞预检 | 开/关 | 开 | 用正则检测 `malloc(a*b)` 无溢出保护、`arr[i+1]` 无边界检查等。**零 token，强烈建议开启** |

**使用建议：**

- 测试 BigVul/Juliet → 无sink 判安全，省 token
- 测试 D2A/PrimeVul → 无sink 送 LLM，防止漏掉逻辑错误
- 消毒器阈值设 2 通常足够，设太高会误判安全

---

### 3.3 代码窗口

**作用：** 控制送入 LLM 的代码量。代码越长 → LLM 上下文越完整 → 但 token 消耗越大。

| 参数 | 可选值 | 默认值(v3) | 说明 |
|------|--------|-----------|------|
| 窗口模式 | 简单截断 / IRIS聚焦 / 完整代码 | 简单截断 | 见下方详解 |
| 截断长度 | 500 ~ 8000 字符 | 1000~3000 | 仅简单截断模式生效，取代码前 N 个字符 |
| IRIS窗口 | 1 ~ 20 行 | 5 | 仅 IRIS 模式生效，以发现点为中心 ±N 行 |

**三种模式对比：**

| 模式 | 原理 | Token 消耗 | 适用场景 |
|------|------|-----------|---------|
| 简单截断 | `code[:N]` 取前 N 字符 | N 越大越多 | **推荐。** 500-1000 字符通常足够 LLM 判断 |
| IRIS聚焦 | 围绕 code_patterns 行号 ±N 行 | 取决于代码密度 | 漏洞集中在特定行时精准省钱 |
| 完整代码 | 整段代码全部送入 | 最高 | 逻辑错误需要全局上下文时 |

---

### 3.4 LLM 调用策略

**作用：** 决定 LLM 以什么方式分析代码。这是对 Token 消耗影响最大的参数组。

#### 调用模式

| 模式 | LLM调用次数 | Token 消耗 | 耗时 | 原理 |
|------|-----------|-----------|------|------|
| 单次结构化输出 | **1** | 低 | 快 | 一次 prompt 直接出 JSON 判定，简洁高效 |
| Agent链(筛查→验证) | 2-3 | 中 | 中 | Agent1 快速筛查 → 可疑的送 Agent2 深度验证 |
| 多温度投票 | 4-5 | **高** | 慢 | Agent1 筛查 → Agent2 用 3 个不同温度各跑一次，投票决定 |

#### Agent链 模式参数

| 参数 | 范围 | 默认值 | 说明 |
|------|------|--------|------|
| Agent1 温度 | 0.0 ~ 1.0 | 0.0 | 筛查阶段温度。0.0 = 确定性输出，每次一致 |
| Agent2 温度 | 0.0 ~ 1.0 | 0.1 | 验证阶段温度。稍高让模型多角度审视 |
| Agent2 判定倾向 | 宁可误报 / 宁可漏报 | 宁可误报 | **flag_it**：不确定时判漏洞（减少漏报）；**precision**：不确定时判安全（减少误报） |
| Agent3 证据收集 | 开/关 | 关 | 开启后额外调一次 LLM 收集 CWE 编号、行号和修复建议。**+1 次调用** |

#### 多温度投票 模式参数

| 参数 | 范围 | 默认值 | 说明 |
|------|------|--------|------|
| Agent1 温度 | 0.0 ~ 1.0 | 0.0 | 同上 |
| 投票共识 | 1 ~ 3 | 2 | ≥N 票判漏洞才确认。设 1 = 有 1 票就算漏洞；设 3 = 全票通过才确认 |
| 加权投票 | 开/关 | 关 | 开启后低温(0.0)票权重 1.5，高温(0.3)票权重 1.0。低温更"清醒"更可信 |
| Agent3 证据收集 | 开/关 | 关 | 同上 |

#### 单次调用 模式参数

| 参数 | 范围 | 默认值 | 说明 |
|------|------|--------|------|
| 温度 | 0.0 ~ 1.0 | 0.3 | 输出随机性。0.0 更快更短，0.3 给推理留空间 |
| 最大Token数 | 256 ~ 4096 | 2048 | 限制 LLM 输出长度。JSON 响应通常 100-200 token，设 1024 足够 |

---

### 3.5 后处理

**作用：** LLM 输出后的二次校验。纯静态分析（正则），**零 token 消耗**。

| 参数 | 说明 | 建议 |
|------|------|------|
| 安全模式检测 | LLM 判漏洞 → 用正则检查是否实际安全（如 memcpy 用了 sizeof、printf 用了字面量格式串）→ 覆盖为安全 | **开启**，过滤 LLM 误报 |
| 数值漏洞兜底 | LLM 判安全 → 用正则检查是否有整数溢出/Off-by-one → 覆盖为有漏洞 | **开启**，弥补 LLM 漏掉的数值漏洞 |

---

### 3.6 JSON 解析器

| 模式 | 原理 | 建议 |
|------|------|------|
| 鲁棒模式(截断修复) | LLM 输出被截断时，尝试从残片中恢复 JSON 键值对 | 默认推荐，容错性好 |
| 简单模式 | 标准 JSON 解析 + 简单括号补全，失败就放弃 | 略快一点 |

## 四、温度参数详解

温度控制 LLM 输出的**随机程度**，范围 0.0 ~ 1.0。

```
温度 = 0.0  →  每次问同一个问题，答案完全一样（确定性）
温度 = 0.3  →  答案基本稳定，措辞可能微调
温度 = 1.0  →  每次答案都可能不同，更有"创意"
```

**底层原理：** LLM 对每个候选 token 算概率分布，温度缩放这个分布。低温让高概率 token 更突出，高温让低概率 token 也有机会被选中。

**实际影响：**

- **温度 0.0**：输出最短、最稳定。适合筛查（Agent1），同一段代码每次判定一致
- **温度 0.1-0.3**：轻微随机性，适合验证环节或单次调用
- **温度 0.7-1.0**：多温度投票中用作"不同视角"，产生有差异的判断

**省 Token 技巧：** 温度设 0.0 时模型走最短路径，输出通常比高温度少 20-30%。

## 五、推荐配置方案

### 极简测试（最少 Token）

```
预设方案: v3 (单次调用)
静态决策:
  无sink函数 → 判安全          ← 注意：会漏 D2A/PrimeVul
  低风险+sink → 判安全
  消毒器阈值: 2
  数值漏洞预检: ✓
代码窗口:
  模式: 简单截断, 截断长度: 1000
LLM:
  模式: 单次结构化输出
  温度: 0.0
  最大Token数: 1024
后处理:
  安全模式检测: ✓
  数值漏洞兜底: ✓
JSON解析器: 简单模式
```

**预期：~500 token/样本（送LLM的），大量样本被静态决策过滤（0 token）**

### D2A / PrimeVul（逻辑错误不遗漏）

与极简测试相同，仅修改：

```
静态决策:
  无sink函数 → 送LLM            ← 关键！逻辑错误常无sink
  消毒器阈值: 0                  ← 逻辑错误不一定有消毒器
```

### 全面检测（追求准确率）

```
预设方案: v1 (IRIS)
静态决策:
  无sink函数 → 送LLM
  低风险+sink → 送LLM
  消毒器阈值: 0
  数值漏洞预检: ✓
代码窗口:
  模式: IRIS聚焦, IRIS窗口: 5
LLM:
  模式: Agent链
  Agent1 温度: 0.0
  Agent2 温度: 0.1
  Agent2 倾向: 宁可误报
  Agent3: ✓
后处理:
  安全模式检测: ✓
  数值漏洞兜底: ✓
```

## 六、测试流程

1. 启动：`streamlit run streamlit_app.py`
2. 侧栏顶部 → 填写 API Key 和模型名 → 点击「测试连通性」
3. 看到「已连通」→ 选择数据集和样本序号
4. 根据数据集类型调整「无sink函数 →」的选项
5. 点击「▶ 运行管道」→ 观察各层输出
6. 查看最终判定是否与真实标签一致

**快速验证三件套：** 选 BigVul 样本 0 → 预设 v3 → 跑通 → 切 v1/v2 对比 → 切自定义微调参数
