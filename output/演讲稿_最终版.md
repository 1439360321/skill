## 开场（PPT 1-2，约 1 分钟）

**（PPT 1 — 封面）**

各位老师、同学，下午好。我们组做的题目是"基于大语言模型的应用安全审计技术"。核心思路一句话概括：把静态分析工具和大语言模型结合起来检测代码漏洞——零样本、不微调。

## 一、核心成绩（PPT 3-6，约 2.5 分钟）

**（PPT 3 — 核心成绩）**

先给大家看最终成绩。在 PrimeVul 数据集上，200 个样本分层测试，F1 达到 0.77，Precision 0.78，Recall 0.75，误报率 0.21。ROC-AUC 是 0.8356，PR-AUC 是 0.7515——说明模型在不同阈值下都有比较稳定的表现。

细分来看：有 sink 的子集——就是代码里有危险函数调用的那 19 个样本——F1 做到了 0.88。没有 sink 的 181 个样本 F1 是 0.74。注意，无 sink 样本占了 90.5%，是整体性能的主要驱动因素，说明我们的方案不是在少数简单样本上刷出来的。

BigVul Clean 数据集上 200 样本，F1 是 0.67。这里先提一句，BigVul 的标签噪声率学术界估计有 75%，所以 0.67 这个数字可能已经接近数据集的理论上限了——后面局限性部分会展开讲。

**（PPT 4 — 学术对比）**

这几个数字横向比是什么概念？StarCoder2 7B 零样本 F1 只有 0.03——基本上不能用，说明通用代码模型直接做漏洞检测是完全不可行的。CodeBERT 微调后做到 0.21——即使有标注数据专门训练，单一模型的能力也非常有限。LLMxCPG 结合了代码属性图，做到 0.62 到 0.68——结构化表示确实能提供有价值信号。我们的方案在同等条件下达到 0.77，在零样本、纯代码输入、没有微调的设定下超过了所有已知基线。

**（PPT 5 — 成绩汇总）**

这张表汇总了所有数据集上的成绩。除了 PrimeVul 和 BigVul，我们还在 Juliet 上做了 30 个样本的跨数据集验证——F1 0.68，证明架构不是对单一数据集过拟合。

**（PPT 6 — 平台概览截图）**

我们最后做成系统网站。

这是我们系统的首页仪表板。左边可以看到模型状态、可用数据集数量；右边是快捷操作入口——代码扫描、批量评测、管道配置，后面讲架构的时候会对应到具体页面。

## 二、Phase 1：起点 —— sink 注册表过拟合与四个优化层的失效（PPT 7-15，约 4 分钟）

**（PPT 7 — Phase 1 分隔页）**

好，成绩讲完，我们来回顾一下是怎么走到这里的。Phase 1 是我们开始的地方。

**（PPT 8 — 初始框架）**

最初搭的基础架子很简单：tree-sitter 做多语言 AST 解析，一个 sink 危险函数注册表——就是预定义的 strcpy、memmove、system 等危险函数列表，以及 Ollama 本地部署的模型来做漏洞推理。系统界面支持了三种 LLM 接入方式——GLM-4.6V、DeepSeek V4 Flash、Ollama 本地部署，API Key 通过 .env 文件加载，不会提交到 Git。

**（PPT 9 — LLM 配置截图）**

这是我们的 LLM 配置页面，可以看到三种模型提供商的切换。后来我们从 Ollama 的本地 deepseek-r1 8B 切换到了云端的 GLM-4.6V，因为本地 8B 模型对安全审计任务来说能力确实不够。

**（PPT 10 — Sink 注册表数据泄漏）**

这里我要做一个方法论上的坦诚交代。sink 注册表——这个危险函数列表——是在一个早期的自建测试集上编写和调优的。当时对数据泄漏问题意识不足，直接在测试样本上观察哪些函数出现频繁、哪些跟漏洞标注有关联，然后加进去。这本质上是在那个早期测试集上"训练"了 sink 规则。

意识到问题后我们立刻换了评测数据集——后续所有正式评测都用 PrimeVul 和 BigVul，跟编写 sink 规则的数据集完全独立。客观来说，sink 函数本身是通用编程知识——strcpy 就是危险、system 就是可注入——这些跟具体数据集无关。泄漏的影响是方向性而非持续性的。

我们选择坦诚讲这个问题，是因为学术诚信——方法的局限性和犯错的地方，都应该说清楚。

**（PPT 11 — 四个优化层）**

在 sink 注册表之上，我们又叠加了四个"优化层"。数据流追踪在函数内做 Source→Sink 路径分析。Sanitization 检测用正则表达式检查安全措施。三阶段 LLM 推理——筛查、深度分析、自检。RAG 知识库用 BigVul 案例做检索增强。

在一个 59 样本的自建集上 F1 超过 0.9，看起来很好。但接下来我们会看到，三个优化层几乎全部失效了。

**（PPT 12 — 失效① 数据流追踪）**

数据流追踪被做成了硬过滤器——没有数据流路径就直接判安全。但致命的问题是：函数参数天然就是外部输入来源，数据流分析几乎总能连上——Source→Sink 路径几乎总是存在。硬过滤条件没有区分度，等价于随机过滤。更糟糕的是，它反而砍掉了 17 个包含真漏洞的代码切片——TP 被系统性漏掉了。

修正方案很直接：改成纯建议模式，不参与硬判决。数据流信息只作为 Agent 推理的辅助参考。

**（PPT 13 — 失效② Sanitization）**

Sanitization 是另一个典型问题。10 个正则表达式，6 个跟真实代码对不上。左边是我们写正则时假设的理想代码——简单的 if (len > MAX) return。右边是实际遇到的真实代码——有宏包裹、嵌套调用、条件编译，正则完全匹配不到。

而且检测到了 sanitization 不等于真的安全。sizeof 对变长指针不提供任何保护——在 x86-64 上 sizeof(ptr) 永远返回 8。strncpy 不保证空终止——如果源长度大于等于 n，目标不会以反斜杠零结尾。

修正方案：sanitization 检测保留在 CodeSlicer 中，但只影响风险评级不做硬拦截。

**（PPT 14 — 失效③④）**

三阶段 LLM 推理对 F1 的贡献只有千分之十一——0.8657 到 0.8768，基本等于没贡献。deepseek-r1 8B 对安全代码过度推理，单文件跑 40 到 120 秒，完全不可扩展。

59 样本上 0.9+ 的成绩是过拟合的——数据集太小没有统计区分度。但留下了一个重要的正确判断：LLM 的角色是"发现"而不是"验证"——应该弥补静态工具的盲区，比如逻辑漏洞、UAF、竞态条件，而不是重新发明静态工具已经做好的事。

**（PPT 15 — Phase 1 教训）**

Phase 1 的关键教训总结成四句话：不要堆砌优化层，不要在测试集上写规则，不要用静态分析做硬判决，LLM 的目标是补偿盲区。

## 三、Phase 2：模块化 —— LLM 层层杀 TP 与静态拦截（PPT 16-22，约 3.5 分钟）

**（PPT 16-17 — 模块化架构 + 四种模式）**

第二阶段我们把管道拆成可配置的四层模块化架构——静态决策层、代码窗口层、LLM 策略层、后处理层，每层参数独立调节。同时引入了四种 LLM 策略模式。

模式一是单次推理——一次 LLM 调用，token 消耗最低。模式二是三级 Agent 链——筛查、复审、盲区扫描。模式三是多温度投票——三个温度并行推理然后加权投票，利用温度变化产生推理多样性。模式四是工具感知链——这是最终采用的方案。

**（PPT 18 — LLM 层层杀 TP）**

这个阶段做了关键实验。BigVul 52 样本上揭示了一个核心矛盾：LLM 管道层层杀 TP，不增 TP。静态工具找到 24 个真漏洞，Agent1 思维链筛查砍掉 5 到 6 个，Agent2 复审查验再砍 3 到 8 个，最终 TP 从 24 降到 10 到 19。LLM 的价值全部体现在削减误报上——它不能发现新漏洞。

多温度投票的共识阈值设为"至少两票"也太保守了——有漏洞但一个温度判安全就不敢判。

核心启示：Agent 链不能设计为层层审批。每多一层审批，就多一道杀 TP 的风险。Agent 的角色应该是互补而非串行审批。

**（PPT 19 — 代码窗口）**

还有一个反直觉的发现：代码窗口不是越大越好。截取前 1500 字符 F1 是 0.60，截取前 3000 字符反而降到 0.42。原因：函数签名和变量声明是最关键的上下文——它们告诉 LLM 变量类型、大小和来源。1500 字符刚好覆盖这些关键信息。3000 字符灌入了后面不相关的代码，形成噪声，反而干扰判断。这直接催生了 V4 中的动态窗口策略。

**（PPT 20 — 代码扫描截图）**

这是我们的代码扫描界面。左侧是代码输入区，右侧是 LLM 分析结果——漏洞类型、风险等级、推理过程。这个界面展示了单样本的完整分析流程。

**（PPT 21 — 静态拦截）**

第二个关键实验揭示了更严重的问题——静态决策层在杀死性能。V2 预设配置了 no_sink 等于 safe，30 个样本全部在静态层被拦截，LLM 一次都没运行，F1 直接归零。V3 更极端——三把锁同时锁死，F1 同样归零。

关掉所有拦截后，F1 从 0.00 跳到 0.67。之前 V1 到 V3 报告的低分不是因为 LLM 能力不足，而是静态层根本没让它上场——把主角拦在了门外。

**（PPT 22 — Phase 2 关键发现）**

Phase 2 教会我们：不要用静态分析代替 LLM 的判断。LLM 管道中每一层都在杀 TP，Agent 的角色应该是互补而非串行审批。

## 四、Phase 3：V4 三级 Agent + 工具感知链（PPT 23-31，约 4.5 分钟）

**（PPT 23-24 — V4 管道总览）**

在所有这些经验教训上，我们做了架构重建。V4 的工具感知链分七个阶段：工具聚合、共识分析、Agent1 差异分析、有 sink 走 Agent2 激进验证、无 sink 走 Agent2 checklist 审计、Agent3 盲区扫描、后处理校准。核心设计原则：静态分析只提供信号不做硬判决，所有样本都必须经过 LLM 推理。

**（PPT 25 — 工具聚合层）**

工具聚合层并行运行三个静态分析工具。CodeSlicer 是我们自研的 tree-sitter 分析器——零依赖、毫秒级、精确到行。CodeQL 是 GitHub 的企业级引擎——做跨函数污点追踪，检测到了 CodeSlicer 漏掉的 UAF 和整数溢出。Semgrep 是快速模式匹配——极快、规则可定制。

三个工具输出归一化后做共识分析：两个以上工具在同一区域共识就是高信号，单工具发现是中等信号，全部沉默就是无信号——需要盲区扫描补充。

**（PPT 26 — Agent1 设计）**

Agent1 的设计有一个关键原则：只看工具报告，不看原始代码。为什么？第一，如果 Agent1 先看代码就会形成预判，失去交叉验证的意义——它的角色是差异分析师不是初级审计员。第二，工具报告只有几百个 token，远小于完整代码，极大节约上下文窗口。

Agent1 输出三个关键决策：initial_verdict 决定是否继续审查，window_suggestion 决定代码窗口大小，blind_spot_risk 标注工具组合可能遗漏的风险类型。

**（PPT 27 — 有 Sink vs 无 Sink）**

这是管道最核心的设计分歧——有 sink 和无 sink 走两条完全不同的路径。

有 sink 路径，Agent2 用激进验证策略。三条原则：假设 sink 始终可达除非硬证据证明不可达，对清理措施做有罪推定——sizeof 对变长数据不保护、strncpy 不保证空终止，只有证明漏洞不可能时才判 safe。为什么这么激进？安全审计场景下漏报代价远大于误报。

无 sink 路径，所有工具沉默了但不代表代码安全。UAF、竞态、off-by-one 根本不对应任何 sink 模式。这时用 checklist 六类专项审计——让 LLM 逐项填表：MEMORY 项 UAF=YES/NO 证据在第几行，而不是问"可能有漏洞吗"。消融实验中，通用扫描 F1=0.62，checklist 六类审计 F1=0.74，提升 19%。

**（PPT 28 — Checklist 六类）**

六类的划分基于 CWE Top 25 和 OWASP 分类。内存安全——UAF、双重释放、悬垂指针。整数安全——溢出、截断、符号转换。边界安全——数组越界、off-by-one。并发安全——竞态条件、TOCTOU、死锁。错误路径——空指针、资源泄漏、返回值忽略。逻辑缺陷——条件恒真假、分支缺失、索引越界。逐项排查，消除模糊空间。

**（PPT 29 — Agent3 盲区扫描器）**

Agent3 只在 Agent2 判 safe 或 uncertain 时触发，触发率从短函数 0% 到长函数 20%。它的职责是只找漏洞不判安全——发现了就补进结果，没发现也不代表代码安全。

但有个意外发现：A3 加仲裁实际上在架空 Agent2。Agent2 判安全后，A3 重新扫描通常能找到遗漏的漏洞，仲裁器随后推翻 A2。这意味着 Agent2 的倾向性被 A3 完全覆盖了，最终判决由 A3 主导。这是一个需要进一步研究的架构问题。

**（PPT 30 — 后处理层）**

后处理层三个模块零 LLM 成本。冲突仲裁器在 A2 和 A3 结论不一致时用三条规则判决。置信度校准器是纯规则引擎——结论一致加分、模糊措辞减分、多工具共识微调。输出质量检查器捕获三种异常——无理由判断、过度自信、不该犹豫时犹豫。

## 五、消融实验（PPT 31-35，约 2 分钟）

**（PPT 31-32 — 消融结果表）**

控制变量法消融——30 个分层样本，每次只改一个变量，其余跟 V4 保持一致。基线单次 LLM 调用 F1 就是 0.40——说明完整代码加清晰 prompt 已经是强基线。V4 三层 Agent 提升到 0.62，加 55%。关闭 RAG F1 完全不变，说明 RAG 零贡献。多温度投票 F1 持平但调用多了三分之一。单次调用加静态拦截 F1 暴跌到 0.13。

**（PPT 33 — 评测界面截图）**

这是我们的批量评测界面。选择数据集、预设配置后，可以看到评测进度和最终 F1、Precision、Recall 结果。刚才表格里的数据就是在这个页面跑出来的。

**（PPT 34 — 消融结论 ①②）**

RAG 零贡献。我们搭了完整管线——SiliconFlow bge-m3 嵌入、ChromaDB 向量库、NVD 25 万条加 CISA KEV 1647 条共 5.9 万文档——但关闭 RAG 后 F1 完全不变。根因是 CVE 元数据描述漏洞影响不包含代码模式，通用文本嵌入在代码和漏洞之间匹配失败。

静态拦截是最大的人为损害。打开 no_sink=safe 后 F1 从 0.40 降到 0.13，28 个样本只调用了一次 LLM，其余 27 个被拦截。静态层把 90% 的样本判为安全，LLM 根本没上场。

**（PPT 35 — 消融结论 ③④⑤）**

多温度投票换不来提升——DeepSeek V4 Flash 的温度变化不足以产生有效推理多样性。单次 LLM 就是强基线——完整代码加清晰 prompt 能达到 0.40，超过所有带静态拦截的配置。A3 加仲裁在架空 Agent2——后期 Agent 覆盖而非协作前期 Agent 的判决。

## 六、局限性（PPT 36-38，约 1.5 分钟）

**（PPT 36-37 — 局限性 ①②③）**

坦诚讨论六个局限。RAG 没有实际贡献——方向对但嵌入方案选型不对。数据流仅限单函数——跨函数漏洞模式完全看不到。Token 消耗大——三层架构约三倍成本。测试集标签噪声严重——BigVul 估计 75% 的标签是错的，F1=0.67 可能已接近理论上限。仅在函数级片段测试——缺少项目级验证。数据集分布决定组件贡献——PrimeVul 90.5% 无 sink，真正驱动提升的是 checklist 审计。

**（PPT 38 — 局限性 ④⑤⑥）**

标签噪声这个问题值得特别强调。PrimeVul 里 93% 的漏洞样本 CWE 字段为空，人工验证发现 Chrome 的代码重构和变量重命名被标成了漏洞。如果四分之三的标签是错的，再好的模型也难突破。改进标签质量是未来工作的重要方向。

---

## 七、总结（PPT 39-40，约 1 分钟）

**（PPT 39 — 迭代历程）**

这张表回顾了四个版本的关键转变。从 V1 盲目堆砌优化层，到 V2 模块化拆解发现 LLM 只削误报不增 TP，到 V3 静态拦截 F1 归零的惨痛教训，到 V4 工具感知链最终打到 0.77。每个版本的失败都在教我们下一步该往哪走。

**（PPT 40 — 总结）**

三句话总结这个项目：不堆砌——四个优化层变成模块化拆解，每个组件独立验证。不拦截——静态拦截 F1 归零就完全放权 LLM，静态分析只提供信号。不猜测——通用 prompt 换成六类 checklist，逐项填表消除模糊空间。

最终成绩 PrimeVul F1 等于 0.77，零样本条件下超过微调基线。谢谢大家，请各位老师批评指正。
